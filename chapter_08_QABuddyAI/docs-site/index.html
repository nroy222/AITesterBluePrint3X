1 Overview

QABuddy.ai is a multi-source Hybrid RAG (Retrieval-Augmented Generation) system purpose-built for QA engineers. It ingests 10 distinct knowledge sources — from test automation frameworks and JIRA tickets to PRDs and Jenkins logs — and answers questions with cited, grounded answers.

The system uses a dense + sparse hybrid search pipeline powered by BAAI/bge-m3 embeddings stored in Qdrant, fused via Reciprocal Rank Fusion (RRF), re-ranked by a bge-reranker-v2-m3 cross-encoder, and finally answered by a Groq-hosted 120B open-weight LLM.

🎯 Purpose

One question → one cited answer grounded in 10 QA knowledge sources. Eliminate context-switching between JIRA, test reports, PRDs, and logs.

⚡ Key Capabilities

Answer questions, generate test cases, perform root cause analysis (RCA) on failures, and review test coverage gaps — all with inline citations.

🔧 Stack

Flask (Python 3.13) · BGE-M3 embeddings · Qdrant vector DB · bge-reranker-v2-m3 · Groq LLM (openai/gpt-oss-120b) · Gunicorn

3.2 Configuration System

The application uses a two-layer configuration approach. Secrets (API keys) are stored in .env and loaded via python-dotenv. Tunable parameters like chunking sizes and retrieval settings are in config.yaml. The config.py module merges both and provides a cfg(path, default) helper for nested YAML access. 

# config.yaml (excerpt)
chunking:
  code:
    max_chars: 2000        # one method/class unit
    min_chars: 250         # tiny units merge into neighbors
  testcases:
    size: 1400             # 1 row = 1 chunk
  jira:
    size: 4800             # 1 ticket = 1 chunk
  docs:
    size: 2048             # ~512 tokens
    overlap: 300

retrieval:
  top_n_hybrid: 20         # candidates per dense / sparse search
  rerank_candidates: 12    # fused list fed to cross-encoder
  top_k: 6                 # final chunks sent to the LLM
  relevance_threshold: 0.22  # max rerank score below this → "not in KB"

3.3 Ingestion Pipeline

The ingestion pipeline processes each of the 10 data sources through four stages:

  1. Load — Each source has a dedicated loader (CSV, PDF, JSON, Markdown, TypeScript, Java, LOG). Loaders parse raw files into structured documents with metadata (source type, path, URL, line numbers).
  2. Chunk — Documents are split into chunks using per-source strategies. Code files are split at method/class boundaries. Test cases are one row per chunk. JIRA tickets are one ticket per chunk. PDFs and docs use size+overlap splitting. Logs extract failure blocks with surrounding context.
  3. Embed — Each chunk is embedded by BAAI/bge-m3, which produces both a dense vector (1024 dimensions) and a sparse (lexical) vector in a single forward pass. This enables hybrid search without running two separate embedding models.
  4. Store — Chunks with their dense and sparse vectors are stored in Qdrant using named vectors. An idempotent manifest (.ingest_manifest.json) tracks file hashes so only changed files are re-embedded on subsequent runs.

3.4 Retrieval Pipeline (Ask)

When a user asks a question, the retrieval pipeline executes a sophisticated multi-step process:

  1. Mode Detection — The question is classified into one of four modes: answer (default), generate (create test cases), rca (root cause analysis), or review (coverage gaps). This is done via regex pattern matching on the question text.
  2. Condense — If conversation history exists, the question is rewritten into a standalone query by resolving pronouns and references from the conversation context using the LLM.
  3. Query Rewrite — The condensed question is expanded into 3 alternate phrasings using the LLM. This improves recall by capturing different ways the same information might be expressed in the knowledge base.
  4. Hybrid Search — All 4 queries (original + 3 rewrites) are embedded with BGE-M3. Each query's dense vector searches Qdrant for the top-20 dense matches, and each sparse vector searches for the top-20 sparse (BM25) matches. Results are deduplicated by ID.
  5. RRF Fusion — The dense and sparse result sets are fused using Reciprocal Rank Fusion (RRF, k=60). This combines the ranking signals from both retrieval methods into a single ranked list.
  6. Rerank — The top-12 fused candidates are re-scored by bge-reranker-v2-m3, a cross-encoder that computes relevance scores between the question and each chunk. The top-6 chunks are kept for the LLM.
  7. Relevance Threshold — If the highest reranker score is below 0.22, the system returns "not in knowledge base" instead of hallucinating an answer.
  8. Prompt Building — A mode-specific system prompt is combined with the top-6 chunks (with source metadata) and conversation history to form the LLM context.
  9. LLM Inference — The prompt is sent to Groq hosting openai/gpt-oss-120b (a 120B open-weight model). The response is streamed back via Server-Sent Events (SSE) with inline citations referencing source files and line numbers.

3.6 LLM Integration

The LLM client (app/llm.py) is provider-agnostic via the OpenAI-compatible API format. By default it uses Groq for ultra-fast inference on openai/gpt-oss-120b, but can be swapped to any OpenAI-compatible provider (OpenRouter, OpenAI, local Ollama) by changing two environment variables: 

LLM_BASE_URL=https://api.groq.com/openai/v1
LLM_MODEL=openai/gpt-oss-120b

The client supports both streaming (chat_stream for SSE) and non-streaming (chat for one-shot) modes. API key resolution checks LLM_API_KEY, GROQ_API_KEY, and OPENROUTER_API_KEY in order.

4 Technologies & Implementation Details

🐍 Python 3.13 + Flask

Backend server with SSE streaming for real-time chat and ingestion progress. Gunicorn for production multi-threaded deployment.

🧠 BAAI/bge-m3

Multi-lingual, multi-function embedding model producing both dense (1024d) and sparse (lexical BM25) vectors in one forward pass. Enables hybrid search without dual encoders.

🗄️ Qdrant

Vector database with native support for named vectors (dense + sparse per point). Supports hybrid search with payload filtering by source type. Runs embedded locally or as a server in Docker.

⚡ bge-reranker-v2-m3

Cross-encoder reranker that computes precise relevance scores between query and candidate chunks. Significantly improves precision over embedding-based similarity alone.

🔄 Reciprocal Rank Fusion

Combines dense and sparse search rankings using the formula: RRF(d) = 1/(k + rank(d)). Robust to score distribution differences between the two retrieval methods.

☁️ Groq LPU Inference

Ultra-fast LLM inference via Groq's Language Processing Units. Hosts openai/gpt-oss-120b (120B parameters) with streaming token generation at <100ms latency.

4.4 Deployment

The application is designed for deployment on a DigitalOcean Droplet using Docker Compose with three services: Qdrant (vector DB), the Flask app (gunicorn), and Caddy (TLS termination + basic auth). The Dockerfile installs Python 3.13, the BGE-M3 model (cached for faster startup), and all dependencies. 

# docker-compose.yml services:
#   qdrant  — vector database (port 6333)
#   app     — gunicorn Flask server (port 5080)
#   caddy   — reverse proxy with TLS + basic auth (port 443)

For local development, Qdrant runs embedded within the Flask process (no separate server needed). The ingestion pipeline uses an idempotent manifest to track file changes, enabling incremental re-ingestion without reprocessing unchanged files.
